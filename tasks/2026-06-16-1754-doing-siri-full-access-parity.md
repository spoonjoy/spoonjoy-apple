## Web Route Classification Contract

Unit 20 owns a generated route manifest sourced from `/Users/arimendelow/Projects/spoonjoy-v2/app/routes.ts`, not a hand-maintained subset. The manifest must flatten every current React Router route module, preserving nested paths and dynamic segments, and each manifest row must include `routeId`, `routeFile`, `urlPattern`, exactly one category, and an explicit `publicShareable` flag that defaults to `false`.

- `native-ui-universal-link`: current product surface opened natively and eligible for AASA when public or authenticated app navigation supports it.
- `secure-web-handoff`: existing security, auth, OAuth, API credential, agent-connect, or approval flow that should open in the app as a guarded web handoff instead of a native reimplementation.
- `api-or-oauth-non-aasa`: API, MCP, WebAuthn, OAuth metadata/token/revoke/register, push, or REST endpoint paths that native may call but should not be user-facing AASA navigation.
- `platform-asset`: platform or metadata assets such as OG images, photos, health, well-known devtools JSON, and well-known OAuth metadata.
- `intentional-exclude`: catch-all, framework fallback, or route that has an explicit reason not to be native UI, secure handoff, or AASA.

The manifest must fail validation on any unclassified route module, duplicate route module entry, route module present in `app/routes.ts` but absent from the manifest, manifest route module absent from `app/routes.ts`, or `publicShareable: true` on a non-public route. Layout/index sibling modules may share one `urlPattern` only when all rows for that URL coalesce to the same URL-level AASA/share decision; `/recipes` and `/cookbooks` must have explicit tests for this layout/index coalescing. Public URL builders may target only routes with `publicShareable: true`; current public shareable objects are recipes and cookbooks. Shopping lists/items, spoons/cook logs, capture drafts/imports, settings, account/security routes, and provider/API handoffs may expose native transfer values or secure/private in-app destinations, but must not generate public `spoonjoy.app` URLs unless a real web route and tests are added first.

Freshness thresholds are fixed for implementation and tests: signed-in bootstrap/account/settings/shopping data is fresh for 15 minutes after `lastValidatedAt`; recipe/cookbook detail, spoon lists, profile/chef graph, and active cook-mode backing data are fresh for 6 hours; catalog/search result pages are fresh for 24 hours; local cook progress, queued mutations, capture drafts, and staged media are locally authoritative until synced, discarded, or conflicted. The app must attempt revalidation on launch, foreground, account/environment change, network recovery, and when a visible surface opens with stale data. If revalidation fails, the app keeps serving cache with the matching stale/offline/sync-failed indicator state.

Offline queue policy is fixed for implementation and tests:

| Queueable while offline | Online-only while offline |
| --- | --- |
| Recipe create/update/delete/fork; step create/update/delete/reorder; step ingredient add/delete; dependency edits; cookbook create/rename/delete/add/remove recipe; shopping item add/check/delete/add-from-recipe/clear-completed/clear-all; cook progress/checkoff/timer/scale changes; spoon create/update/delete/photo payload updates; cover upload/set-active/remove/archive/regenerate/from-spoon; capture draft create/edit/discard; import submit; profile display-field update; profile photo upload/remove after local media staging; notification preference update; APNs device registration/revocation after a device token exists; Siri writes that map to one of these queueable mutations. | OAuth sign-in/callback; token create; token revoke; OAuth connection disconnect; logout/session revoke; passkey/password/provider-link actions; APNs permission prompt; APNs device-token acquisition; production AASA/APNs/TestFlight validation; provider-generated cover regeneration or recipe import when the matching `ProviderSecret` blocker exists; destructive production operation approval. |

Queued mutations must include stable `clientMutationId`, endpoint path, method, idempotency key, payload schema version, created-at time, dependency ordering key, retry count, and last error. Queue replay is FIFO within an object dependency key and may run independent keys concurrently only after tests prove no ordering conflict. Retry backoff is 5 seconds, 30 seconds, 5 minutes, then 30 minutes with jitter capped at 20%; auth failure pauses the queue until reauth, validation conflict marks only that mutation conflicted, and server tombstones remove or conflict local records according to the endpoint-specific tests.

Local media staging policy is fixed for implementation and tests: individual staged media files are capped at 25 MB; temporary generated previews are capped at 128 MB total and may be LRU-evicted after successful upload or discard; unsynced user-selected media is capped at 512 MB total or 100 files per account/environment. Unsynced user media is never silently evicted; if the cap is reached, new staging fails with a visible local-storage state that tells the user to sync or delete drafts.

Spotlight and App Intents privacy policy is fixed for implementation and tests: identifiers for private domains must include account, environment, and schema namespace; logout, session revoke, account switch, environment switch, cache deletion, and tombstone/delete replay must purge matching Spotlight entries and donated private shortcuts/entities. Indexed content for shopping, spoons/cook logs, profiles, and capture drafts must include only user-visible summary fields and must exclude bearer/session/token values, provider secrets, raw media paths, raw imported source text when not deliberately user-visible, provider-blocker details, and hidden conflict/debug metadata. Stale private searchable items may remain available only while their owning account/environment cache is valid for display; final validation must prove stale or orphaned private entries cannot survive purge.
